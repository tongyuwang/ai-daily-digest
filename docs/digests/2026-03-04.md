### 2. Grep Is Dead: How I Made Claude Code Actually Remember Things
**[Article on X](https://x.com/i/article/2028328572272742401)** · 👁 793K impressions · 🔖 8,621 bookmarks

The most-bookmarked new piece in this cycle by a wide margin. The core argument: coding agents that retrieve context through keyword search (grep-style) hit a hard ceiling, because the most relevant context is often semantically distant from the search term.

The piece walks through a concrete memory architecture built on top of Claude Code:
- Structured memory files organized by semantic domain, not filename
- Agent-maintained indexes that update after every significant session
- Retrieval that surfaces related context the agent didn't know to ask for

The result: agents that stay coherent across multi-day, multi-file projects — and that don't repeat mistakes they've already made. The 8,600+ bookmarks suggest this is hitting a pain point many teams are actively fighting.

### 3. Every Practical Security Tip I Have on Agents
**[@elvissun](https://x.com/elvissun/status/2028483837496635809)** · 👁 171K impressions · 🔖 2,270 bookmarks

A dense, practical checklist from someone clearly running agents in production. No fluff. The highlights:

- **Security by least privilege, not by prompts** — Don't rely on "you are not allowed to..." instructions in the system prompt. Enforce at the infra level.
- **Billing cap on everything AI touches** — An uncontrolled agent loop is a financial incident waiting to happen.
- **Wrap all external data in `<UNTRUSTED_EXTERNAL_CONTEXT>`** — Treating external inputs as untrusted by default is the cleanest prompt injection mitigation available today.
- **Don't post publicly about what access your agent has** — Those are prompt injection invitations.
- **One-time access for sensitive integrations** (e.g. Gmail) with human-in-the-loop, self-revoke on finish.
- **Audit trails everywhere → feed back into security self-improvements**

The closer: *"Mistakes will happen. Limit worst case, embrace the rest."* A realistic, non-paranoid framing of agent security.

### 4. Evals Skills for Coding Agents
**[Article on X](https://x.com/i/article/2028710814601908224)** · 👁 111K impressions · 🔖 2,610 bookmarks

A sharp piece that reframes eval design from "testing" to a first-class engineering skill. The central insight: most teams treat evals as a binary pass/fail gate, when the real value is in the trace — understanding *why* an agent succeeded or failed, and using that signal to improve the system.

Key ideas:
- Automated correctness labeling (e.g., "did the PR get merged?") as a native signal rather than manual annotation
- Collecting traces at scale and routing them to a separate LLM to identify patterns of failure
- Tracking "correctness progress" as a first-class metric over time

The post from @18708003 adds a key nuance: "The real valuable piece is when step 2 just happens natively during the whole thing, and users don't have to adjust their behavior." Great evals are invisible to users — they happen as a byproduct of normal workflow.

### 5. Karpathy: The Coming Era of Bespoke Software
**[@karpathy](https://x.com/karpathy/status/2024583544157458452)** · 👁 1.88M impressions · 🔖 7,978 bookmarks

Karpathy vibe-coded a custom cardio experiment tracker in one hour — Claude had to reverse-engineer the Woodway treadmill cloud API, process the data, and build a web UI — and used it as a launchpad for a broader thesis that's been ricocheting around the AI community since.

The core argument:
1. **The app store is an outdated concept.** There will never (and shouldn't ever) be an App Store entry for "cardio experiment tracker." For a 300-line app an LLM agent gives you in seconds, the abstraction of a "discrete app you download" is the wrong level.
2. **Services need agent-native ergonomics.** Every connected product is a sensor or actuator. It shouldn't maintain a human-readable frontend — it should be an API/CLI that agents can use directly. *"99% of products/services still don't have an AI-native CLI yet. In 2026. What am I a computer? You do it."*
3. **The 1-hour build should be a 1-minute build.** What has to be true — personal context, skill libraries, sensor integrations — for that to happen?

The reply from @qwibitai (NanoClaw) is also worth reading: a lightweight OpenClaw alternative built with exactly this philosophy in mind.
