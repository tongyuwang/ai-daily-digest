### 3. OpenAI Harness Engineering: 1M Lines of Code, Zero Written by Hand
**[OpenAI Blog](https://openai.com/index/harness-engineering/)** · 👁 1,163 impressions · 🔖 21 bookmarks

Low bookmark count but high importance — this is OpenAI's own engineering blog post on their internal Codex experiment. Over 5 months, a 3-7 engineer team shipped a production product with ~1 million lines of code, all written by Codex (zero manual lines). ~1,500 PRs merged. 3.5 PRs per engineer per day, increasing as the team grew.

Key learnings:
- **Humans steer, agents execute.** The engineering job became: design environments, specify intent, build feedback loops.
- **The repository was shaped by the agent from commit #1.** Even the initial AGENTS.md file was written by Codex.
- **They built in ~1/10th the time** it would have taken to write by hand.

This is a rare, credible data point on what fully agent-driven engineering actually looks like in practice.

### 4. Cursor Automations Launch
**[@ericzakariasson (Cursor)](https://x.com/ericzakariasson/status/2029604478564151577)** · 👁 98K impressions · 🔖 469 bookmarks

Cursor officially launched Automations — the ability to trigger agents based on real-world events: incidents, PRs, Slack messages, or any webhook. The announcement came with examples of internal Cursor automations, including a security audit that runs on every push to main.

A follow-up analysis thread (👁 31K, 🔖 170) unpacks why this matters structurally: *"Engineers can now manage 10+ coding agents at once, but human attention became the bottleneck. You can't babysit a dozen agents while also doing your actual job."* Automations flip the model — instead of humans launching agents, events do.

The four things this unlocks:
1. Continuous code review at a depth humans skip
2. Incident response that starts before you're paged
3. Maintenance on a schedule, not when someone remembers
4. Knowledge synthesis across tools (Slack, Linear, GitHub, Datadog via MCP)

Cursor's own code review lifecycle is now fully automated: low-risk changes auto-approved, medium/high-risk changes get reviewers assigned based on commit history.
